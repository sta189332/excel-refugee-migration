Recommended R Packages by Use Case

This appendix is not a list for curiosity.

It is a working map for people who are already under pressure and need the right package fast. The purpose is not to expose you to every interesting corner of the R ecosystem. The purpose is to stop the package-guessing cycle before it wastes another morning.

If a package appears here, it earns its place because it solves a real problem from this book’s world. Dirty exports. Repeated joins. Pivot-table replacements. Leadership charts. Excel handoffs. Folder chaos. In-session AI help that stays inside RStudio.

That is the standard.

TipCheat Sheet Box

Pick the package by the problem.

  • Import problem: readr, readxl, rio, openxlsx
  • Cleaning problem: janitor, stringr, lubridate, tidyr, dplyr
  • Transformation problem: dplyr, tidyr, forcats
  • Visualization problem: ggplot2, scales, forcats
  • Reporting problem: write_csv(), write.xlsx(), ggsave(), Quarto
  • Project hygiene problem: here, fs, usethis
  • In-session AI problem: ellmer, chattr

C.1 Data Import

Import is the first pressure point in spreadsheet-heavy work. If the file comes in badly, everything after it gets slower, louder, and less trustworthy. So this section is not about “reading data” in a generic sense. It is about choosing the shortest clean path from the file you received to the table you can actually work with.

Data Import Use-Case Map

Use Case Recommended Package Core Function(s) Best For Why It Helps Spreadsheet Refugees Typical Workflow
Import a standard CSV export readr read_csv() Reporting exports, operational data pulls, sales actuals, KPI files Fast, predictable, widely used, and easier to read than older base import syntax Read the CSV, inspect with glimpse(), clean names, start cleaning immediately
Import an Excel workbook readxl read_excel() Finance sheets, HR workbooks, targets, budget files Lets you pull in spreadsheet data without opening Excel first, which reduces copy-paste behavior Read the workbook or sheet, inspect structure, standardize names, then clean
Import mixed file types through one simple interface rio import() When the team sends CSV one week and Excel the next Removes guesswork when source types vary and you just need the file in R quickly Use import() as a fast entry point, then switch into standard cleaning workflow
Import from a specific sheet readxl read_excel(sheet = "...") Multi-sheet finance packs or planning files Stops you from pulling the whole workbook in when one tab is the actual source Import the exact tab, clean names, confirm the right sheet before analysis
Write a clean CSV for downstream use readr write_csv() Clean outputs for analysts, systems, and version-safe exports Keeps outputs light, transparent, and easy to reload later Save cleaned or summarized data after validation
Write back to Excel for stakeholders openxlsx write.xlsx() Managers or teams who still expect .xlsx delivery Lets the engine stay in R while the final handoff stays familiar Build the workflow in R, export the final summary or cleaned file to Excel
Write multiple objects into an Excel workbook openxlsx write.xlsx(list(...)) Multi-tab deliverables for finance or HR review Replaces manual tab-copy routines with a scripted export Save several validated tables into one workbook with named sheets

What This Package Combination Does Best

readr, readxl, rio, and openxlsx cover almost the entire import-and-return cycle most spreadsheet refugees live inside.

readr is the first reach when the source is a stable CSV export. It is fast, predictable, and well-behaved in modern analyst workflows. If a system exports monthly actuals, weekly operations files, staffing data, survey responses, or budget snapshots as CSV, read_csv() is usually the right starting move.

readxl matters because real organizations still run on workbooks. Targets arrive in Excel. Finance assumptions live in Excel. HR planning sheets live in Excel. Budget models live in Excel. You need a package that reads those files without turning the whole workflow back into copy-paste theatre. That is what read_excel() does.

rio is the convenience package for mixed-file environments. It is especially useful when the team is inconsistent and the analyst does not want to keep changing import syntax depending on whether the file this week is .csv, .xlsx, or something else. It is not mandatory. It is useful when file-type variety itself has become friction.

openxlsx matters at the other end. Most stakeholders are not waiting for a philosophical conversation about why CSV is purer. They want a file they can open, circulate, annotate, or attach to email. openxlsx lets you keep the analytical engine in R while respecting the delivery culture around you.

That combination is enough for most business reporting work in this book. You do not need ten import packages competing for attention. You need a small team of packages that covers raw intake, workbook handling, flexible import, and familiar export.

When To Reach For It

Use readr when the file is a CSV export and the workflow needs speed, simplicity, and low ceremony.

Use readxl when the source is a workbook, when you need a specific sheet, or when the file came from a team that still thinks Excel is the natural home of all business logic.

Use rio when the file type changes unpredictably and you want one mental model for getting data into the session quickly.

Use openxlsx when the final output must go back to Excel because the stakeholder is expecting tabs, workbook structure, or an .xlsx attachment.

A good rule is this:

  • if the file is the source, choose the package that gets it into R cleanly
  • if the file is the deliverable, choose the package that gets it back out professionally
  • if the file type keeps changing, reduce mental friction with rio

This keeps the workflow honest. Import packages are not lifestyle choices. They are problem choices.

Typical Workflow

A real import workflow usually looks like this:

First, bring in the raw actuals from CSV with readr.

Second, bring in the targets or assumptions workbook with readxl, often from a specific sheet.

Third, import the lookup file, whether it is CSV or Excel, with either readr, readxl, or rio depending on the source.

Fourth, inspect all three tables before touching anything. Not because inspection is ceremonial, but because imports lie quietly when column types or sheet choices are wrong.

Fifth, clean names and confirm the key fields are present before any joining begins.

Sixth, once the analysis is validated, export the summary table and, if necessary, a multi-sheet workbook for stakeholder delivery using openxlsx.

That rhythm matters. It keeps import decisions tied to workflow logic instead of file anxiety.

Watch Out For

Reading the wrong sheet is a classic waste of time because the code still runs and the structure can still look plausible. If the workbook has multiple tabs, confirm the tab name or sheet position early.

Do not trust imported column types blindly. A date column may come in as character. A numeric column may come in as text because somebody upstream inserted one non-numeric value. Inspect before you calculate.

Do not export a final workbook before you validate the analysis. If the summary table is still unstable, the pretty Excel file only creates a cleaner version of the wrong answer.

Do not treat rio as an excuse to stop caring about what file you are reading. Convenience is useful. Thought still matters.

Do not keep reopening Excel just to inspect the source structure. Once the file is in R, inspect it there.

Example You Can Adapt

Here is a full import pattern for a typical monthly sales workflow with one CSV, one Excel sheet, one lookup file, and one final Excel export.

library(readr)
library(readxl)
library(rio)
library(janitor)
library(openxlsx)
library(dplyr)

sales_actuals <- read_csv("inputs/sales_actuals.csv") |>
  clean_names()

sales_targets <- read_excel("inputs/sales_targets.xlsx", sheet = "Targets") |>
  clean_names()

product_lookup <- import("inputs/product_lookup.csv") |>
  clean_names()

glimpse(sales_actuals)
glimpse(sales_targets)
glimpse(product_lookup)

monthly_pack <- list(
  actuals = sales_actuals,
  targets = sales_targets,
  product_lookup = product_lookup
)

write.xlsx(monthly_pack, file = "outputs/monthly_sales_import_review.xlsx")

That pattern is not there to impress anyone. It is there to stop the import stage from becoming the first avoidable failure of the day.

WarningQuick Warning

Do not choose the import package by habit alone.

If the file will be handed back to a stakeholder, openxlsx matters. If the source is a stable CSV export, readr is usually enough. If the workbook has five tabs and only one should exist in the workflow, readxl earns its place quickly.

C.2 Data Cleaning

This is the cleanup command center.

If Chapter 6 taught the emotional truth that dirty data creates dirty thinking, this section tells you which packages carry the cleanup work without turning it into a guessing game.

What This Package Combination Does Best

janitor, stringr, lubridate, tidyr, and dplyr work best when they are treated as a stack, not as isolated tricks.

janitor clears the first layer of friction by making column names sane. That sounds small until you have lost ten minutes to a space, a slash, or a system-generated header that turns every later line of code into an irritation.

stringr handles the text problems that spreadsheets quietly normalize. Trailing spaces. Double spaces. Odd capitalization. Labels that are visually similar and operationally different. This is where “West”, “WEST”, and “West” stop pretending they are different regions.

lubridate takes over when dates start humiliating the workflow. Payroll files, targets, monthly KPI packs, and operational exports often arrive with text dates, mixed formats, or reporting periods that need to be derived cleanly. lubridate makes the parsing explicit.

tidyr handles gaps and reshaping problems. Missing values are not just blanks. They are business meaning. replace_na() belongs in the cleaning stack because absence needs to be treated intentionally. separate() and pivot functions matter when the raw file was designed for spreadsheet display rather than analysis.

dplyr is the backbone holding all of it together. It gives the cleaning stack a readable sequence. You are no longer fixing one issue at a time in scattered places. You are building a visible pipeline.

That combination is why cleaning in R starts feeling calmer than cleaning in Excel. The work is still real. The difference is that the fixes start living in one place.

When To Reach For It

Use this cleaning stack when the source file has any of the following:

  • ugly names from a system export
  • spaces and category drift in text columns
  • dates that arrived as text or mixed formats
  • missing values that need business interpretation
  • duplicate rows that need a defined key
  • categories that will fracture a summary if left untouched

That covers most of the real files in this book’s world.

Payroll files need nearly all of it. Staffing extracts usually need names, text cleanup, and duplication checks. Targets and actuals often need name cleanup, date parsing, and category standardization. Monthly KPI packs often need the full sequence because they arrive from several teams, each convinced their naming habits are reasonable.

If the file makes you suspicious before you even start thinking, the cleaning stack is the right first reach.

Typical Workflow

A good cleaning sequence usually runs like this.

Import the file.

Clean the names immediately. Do not let ugly headers into the rest of the pipeline if you can avoid it.

Inspect the structure with glimpse().

Trim and standardize text fields that will be used for grouping or joining.

Parse dates into real dates before attempting time-based summaries.

Handle missing values only after you understand what the blanks mean in business terms.

Check for duplicates using the right business key, not a vague sense that “these rows look similar.”

Review category counts before and after standardization if the field will later appear in a grouped summary.

Export the clean result only after the row count and key totals feel defensible.

That order matters because it reduces accidental distortion.

Watch Out For

Do not replace missing values too early. A blank might mean “not yet assigned,” “not applicable,” or “broken join.” If you replace everything with zero or "Unknown" immediately, you can erase useful diagnostic clues before you understand them.

Do not deduplicate on the wrong key. A duplicate row is not defined by visual resemblance. In payroll, the key might be employee_id plus pay_period. In sales, it might be invoice_id plus product_code. If you deduplicate lazily, you will delete real transactions and feel efficient while doing it.

Do not standardize categories without checking the grouped result. You may think your mapping cleaned the field, then later discover it collapsed distinct business categories into one label because the code was too aggressive.

Do not parse dates with the wrong assumption just because the output looks like a date afterward. A wrong parser produces confident nonsense.

Example You Can Adapt

Here is a full payroll-cleaning recipe with dates, spaces, missing values, duplicates, and export.

library(readxl)
library(dplyr)
library(janitor)
library(stringr)
library(lubridate)
library(tidyr)
library(readr)
library(openxlsx)

payroll_raw <- read_excel("inputs/payroll_extract.xlsx") |>
  clean_names()

payroll_clean <- payroll_raw |>
  mutate(
    employee_id = str_trim(employee_id),
    department = str_trim(department),
    department = str_to_title(department),
    location = str_trim(location),
    pay_date = mdy(pay_date),
    pay_period = floor_date(pay_date, unit = "month"),
    manager_name = replace_na(manager_name, "Unassigned"),
    bonus_amount = replace_na(bonus_amount, 0)
  ) |>
  distinct(employee_id, pay_period, .keep_all = TRUE)

payroll_clean |>
  count(department, sort = TRUE)

write_csv(payroll_clean, "outputs/payroll_clean.csv")
write.xlsx(payroll_clean, "outputs/payroll_clean.xlsx")

This is what a calm cleaning workflow looks like. Nothing mystical. Just a sequence that stops the file from owning the morning.

NotePhD Insight

Cleaning is not a technical prelude. It is part of the analytical claim.

When names, dates, categories, and missing values are inconsistent, the later summary is not merely delayed. It is distorted. That is why cleaning packages belong at the center of the workflow, not at the margins.

C.3 Data Transformation

This is the logic engine of the book.

Transformation is where the work starts feeling less like file rescue and more like analysis. You join tables, create new columns, reshape data, build grouped summaries, and tell the dataset what question it is supposed to answer.

Build calculated columns dplyr mutate() Variance, status flags, group labels, derived metrics Replaces copied formulas with reusable logic Define the business rule once, apply it to the whole column
Group and summarize dplyr group_by(), summarise(), arrange() Regional totals, department counts, monthly performance summaries Replaces recurring pivot-table rebuilds with readable reporting logic Group, summarize, sort, export

What This Package Combination Does Best

Once the file is clean, dplyr, tidyr, and forcats become the core analysis stack because together they cover the three most important jobs in business analysis: relationship, shape, and order.

dplyr handles relationship and calculation. It joins files, builds derived fields, filters rows, groups data, and creates summary tables. It is the package that takes you from spreadsheet gestures to visible logic.

tidyr handles shape. Spreadsheet files are often laid out for human viewing, not for analysis. Month columns stretch across a sheet. Survey answers sit in wide blocks. Budget models treat months like headers. tidyr fixes that by giving you control over long and wide forms when the business question demands it.

forcats handles order. This matters more than people expect. A chart or table that follows alphabetical order when the actual question is ranking, variance, or priority is quietly weakening the analysis. forcats makes the business order explicit.

That combination is powerful because it stays close to the kinds of problems the book keeps solving. You do not need a package explosion to do serious transformation work. You need a stable stack that covers joins, reshaping, summaries, and presentation order.

When To Reach For It

Use dplyr whenever the task sounds like business logic:

  • join actuals to targets
  • calculate variance
  • filter to active staff
  • summarize by region
  • rank departments by headcount
  • count cases by status

Use tidyr when the file shape itself is getting in the way. If the months are columns instead of rows, if the same concept is spread across several headers, or if the final output needs to be widened only after the analysis is done, tidyr is the right reach.

Use forcats when the output will be seen by humans and order matters. If the bar chart should rank from strongest to weakest, if the table should surface the worst variances first, or if the leadership slide should follow business priority instead of alphabetical accident, forcats earns its keep.

Typical Workflow

A strong transformation sequence usually looks like this:

Start from a clean file.

Join the reference table or target table with left_join().

Build the calculated fields with mutate().

Group and summarize with group_by() and summarise().

Reshape only if the business question or final deliverable actually requires it.

Reorder the important categorical field with fct_reorder() if the output is heading toward a chart or ranked table.

Then export or visualize.

This is the point where many spreadsheet users feel the shift most strongly. The output stops being an accumulation of manual actions and starts becoming a readable answer.

Watch Out For

Do not join before cleaning keys. A perfect left_join() on dirty employee_id, region, or product_code fields still produces a weak result.

Do not summarize at the wrong level. Many reporting errors are not math errors. They are level errors. Monthly data summarized across quarters. Department data summarized across the entire organization when the decision needed branch-level detail. Get the grouping level right before admiring the output.

Do not widen too early. Wide tables may feel familiar because they resemble spreadsheets, but long form is usually easier for analysis. Widen only when the final output or stakeholder expectation truly demands it.

Do not reorder categories after plotting as an afterthought. If order matters to interpretation, handle it in the data object before the visual step.

Example You Can Adapt

Here is a full budget-versus-actual workflow using joins, variance metrics, grouped summaries, and final ranking.

library(readr)
library(readxl)
library(dplyr)
library(tidyr)
library(forcats)
library(janitor)

budget <- read_excel("inputs/budget.xlsx") |>
  clean_names()

actuals <- read_csv("inputs/actuals.csv") |>
  clean_names()

budget_actuals <- actuals |>
  left_join(budget, by = c("month", "cost_center")) |>
  mutate(
    budget_value = replace_na(budget_value, 0),
    variance = actual_value - budget_value,
    variance_pct = if_else(budget_value == 0, NA_real_, actual_value / budget_value - 1)
  )

cost_center_summary <- budget_actuals |>
  group_by(cost_center) |>
  summarise(
    total_actual = sum(actual_value, na.rm = TRUE),
    total_budget = sum(budget_value, na.rm = TRUE),
    total_variance = sum(variance, na.rm = TRUE),
    .groups = "drop"
  ) |>
  arrange(desc(total_variance)) |>
  mutate(cost_center = fct_reorder(cost_center, total_variance))

That is the logic engine doing its work. Clean file in. Ranked answer out.

TipCheat Sheet Box

A practical package stack for most corporate analysis work looks like this:

  • dplyr for joins, filters, calculations, and summaries
  • tidyr for reshaping tables
  • forcats when the chart order should follow the business story, not the alphabet

That combination covers more real reporting work than many people expect.

  theme_minimal(base_size = 12)

What This Package Combination Does Best

ggplot2, scales, and forcats are enough for most of the business visuals in this book because the book is not trying to turn readers into gallery curators.

It is trying to help them build charts that do real work.

ggplot2 is the main visual engine. It gives you deliberate control over chart structure rather than default-chart guesswork. scales handles the business language of the chart, which matters more than many people admit. Dollar values should look like money. Rates should look like percentages. Headcount should look like counts, not raw unformatted numbers. forcats gives the chart an order that respects the question instead of alphabetical habit.

That is enough for most reporting work. Ranked comparisons, monthly trends, attrition views, variance charts, departmental summaries, executive slides. You do not need a crowded package stack to do those well. You need clean summaries, clear labels, proper ordering, and disciplined export.

When To Reach For It

Use this stack when the analysis needs to be seen, not just saved.

A ranked comparison needs forcats and ggplot2.

A trend update needs ggplot2 and scales.

An attrition view needs ggplot2, scales, and usually good faceting or grouping logic upstream.

An executive slide needs all three because the chart must be clear, the numbers must be formatted, and the categories must appear in an order that carries the point without help.

This stack is for communication, not decoration.

Typical Workflow

Start with a trusted summary table.

If the chart is categorical, reorder the main category with fct_reorder() so the ranking tells the story correctly.

Build the plot with ggplot() and the right geometry.

Use labs() to state the point clearly, not just the topic.

Format the axis with scales so the numbers speak business language.

Apply a clean theme such as theme_minimal().

Then export with ggsave() at dimensions that will survive a slide deck or report.

That is the chart build sequence worth keeping.

Watch Out For

Do not plot unsorted categories when ranking matters. A chart that leaves the most important bar hidden in alphabetical order is not finished.

Do not overuse legends. If the chart can be understood without one, remove it.

Do not use default titles that say nothing. “Revenue by Region” is technically correct and operationally lazy. The title should tell the audience what matters.

Do not crowd the chart with colors, labels, and ornament because the output feels too plain. Clarity is not emptiness.

Do not export the chart at arbitrary size. An excellent plot can look weak in a deck if it was saved carelessly.

Example You Can Adapt

Here is a full chart recipe that starts with a grouped summary and ends with a leadership-ready PNG.

library(dplyr)
library(forcats)
library(ggplot2)
library(scales)

regional_gap_plot <- regional_gap |>
  mutate(region = fct_reorder(region, total_variance))

gap_chart <- ggplot(regional_gap_plot, aes(x = region, y = total_variance)) +
  geom_col(fill = "steelblue") +
  labs(
    title = "Southeast And North Require Immediate Revenue Attention",
    subtitle = "Cumulative variance against target by region",
    x = NULL,
    y = "Variance"
  ) +
  scale_y_continuous(labels = label_dollar()) +
  theme_minimal(base_size = 12) +
  theme(
    panel.grid.minor = element_blank(),
    legend.position = "none"
  )

ggsave(
  filename = "outputs/regional_gap_chart.png",
  plot = gap_chart,
  width = 8,
  height = 5,
  dpi = 300
)

That is enough for a serious business chart. Not because it is visually noisy. Because it carries the message.

WarningQuick Warning

Do not widen the package stack just because the chart feels important.

Most business visuals in this book do not need a jungle of specialty visualization packages. If the summary table is strong, ggplot2 plus scales plus forcats usually gets you where you need to go.

What This Package Combination Does Best

This combination turns analysis into deliverable without forcing the workflow back into spreadsheet improvisation.

write_csv() is for clean machine-readable handoff. Use it when another analyst, another system, or future-you needs a stable output that can be reloaded without workbook clutter.

write.xlsx() is for the social reality of the office. Some stakeholders want a workbook because that is how they review, comment, circulate, or archive what you send. Exporting to Excel at the end is not regression if the analytical engine stayed in R.

ggsave() matters because charts do not become presentation-ready by accident. A good plot needs dimensions, resolution, and a controlled export.

Quarto matters because repeated reporting is not only about files. It is also about narrative structure. When the workflow grows beyond one summary table and one chart, Quarto becomes the reporting layer that ties logic and communication together.

here() matters because deliverables do not stay credible when they are saved into random folders no one can find later.

ellmer and chattr matter because real delivery work still needs support at the point of friction. If the chart needs one last cleanup or the summary logic feels suspicious, the assistant should sit near the work, not in a separate browser ritual.

When To Reach For It

Keep the deliverable as CSV when the output is mainly for reuse, downstream systems, or another analyst who needs a clean table.

Send it back to Excel when the stakeholder expects a workbook, when multiple tabs matter, or when the audience is not ready to consume anything else.

Use Quarto when the work is becoming a repeatable report rather than a one-off export. If the analysis, chart, and narrative need to stay together in one structured artifact, Quarto is the right presentation layer.

Do not confuse stakeholder preference with workflow truth. The final format can remain familiar while the engine underneath changes completely.

Typical Workflow

A good reporting sequence looks like this:

Validate the summary table first.

Write the clean table to CSV for future reuse.

Export the stakeholder-facing workbook if Excel delivery is required.

Build and save the chart at slide-ready resolution.

If the work needs narrative structure or recurring document output, render it through Quarto.

Keep all outputs inside a stable project path.

That sequence respects both the analytical workflow and the political reality of deliverables.

Watch Out For

Do not export too early. The moment a workbook or chart leaves the workflow, it starts looking official. Make it earn that status first.

Do not save charts at the wrong dimensions and then blame the deck for making them look weak.

Do not treat output format and workflow as the same thing. Sending Excel is not the same as doing the work in Excel.

Do not render Quarto reports before the underlying objects are stable. A polished document can hide unstable logic too well.

C.6 File Paths And Project Hygiene

This is the operational backbone of the book.

A lot of otherwise competent workflows break for reasons that sound small until they destroy an afternoon. Wrong working directory. Broken absolute paths. API keys sitting in scripts. Output files saved into mysterious folders no one can find later. A “quick project” that becomes unmaintainable by the third reporting cycle because nothing was named or structured clearly from the start.

Project hygiene is not glamorous.

It is how work stays repeatable.

What This Package Combination Does Best

here, fs, and usethis protect the workflow from three of the least glamorous and most expensive failures in analytics work: broken paths, file sprawl, and exposed secrets.

here() gives your project a stable center. Files are found relative to the project root, not relative to wherever the working directory happened to be when you clicked something last.

fs gives you a disciplined way to create and maintain structure. If a project needs inputs, outputs, scripts, and archive, create them deliberately. Do not let the project become a desk drawer.

usethis matters because configuration belongs outside the script when it involves credentials. If Gemini is part of the in-session workflow, the key should live in .Renviron, not in code that can be copied, shared, or committed somewhere foolish.

That combination protects your future self from chaos disguised as speed.

When To Reach For It

Use this hygiene stack at the start of a new reporting project.

Use it when work needs to move between laptop and office desktop.

Use it when a recurring workflow has started generating too many “final_v2_use_this” files.

Use it when you set up Gemini inside RStudio and need the key stored safely instead of dangerously.

Use it any time the cost of messy file structure is starting to rival the cost of the analysis itself.

Typical Workflow

Create the project.

Create the core folders with fs.

Use here() for every import and export path.

Store API keys or configuration values in .Renviron with usethis.

Keep names clear enough that another analyst, or future-you after a tired week, can understand the project at a glance.

That is project hygiene in practice. It is not abstract reproducibility theater. It is anti-chaos infrastructure.

Watch Out For

Do not hard-code desktop paths unless you enjoy work that breaks the moment it moves.

Do not keep keys inside scripts. One accidental share, screen capture, or upload and the cleanup becomes much less pleasant.

Do not let folder structure emerge accidentally from whatever files happened to be saved first. Disorder compounds.

Do not use naming conventions that depend entirely on your memory. If “new_final_latest” looks reasonable now, it will not look reasonable next month.

Example You Can Adapt

Here is a complete starter project structure for recurring monthly or weekly reporting.

library(here)
library(fs)

project_name <- "monthly-sales-report"

dir_create(here(project_name))
dir_create(here(project_name, "inputs"))
dir_create(here(project_name, "outputs"))
dir_create(here(project_name, "scripts"))
dir_create(here(project_name, "archive"))
dir_create(here(project_name, "plots"))

Then inside that project:

  • all raw files go to inputs
  • all saved tables go to outputs
  • all reusable scripts go to scripts
  • old source files go to archive
  • all exported figures go to plots

That is enough to stop the project from collapsing under repetition.

Future Package Additions

This section is not filler. It is a controlled expansion rule.

The book deliberately avoids ecosystem overload, but real workflows evolve. When a new package earns consideration, it should pass a stricter test than “someone recommended it online.”

How To Evaluate A Future Package Addition

A future package belongs in this appendix only if it solves a recurring problem better than the current stack, reduces real friction, and stays readable enough for the professional audience of this book.

A package does not qualify just because it is trendy, elegant, or admired by power users.

It qualifies when it improves Monday morning work.

What This Package Does Best

When evaluating a future addition, start by naming the package’s strongest use honestly. Not everything a package can do. The one thing it does well enough to justify learning overhead.

For example, if a package excels at reading specialized files, building publication tables, scheduling scripts, or creating dashboards, say that clearly. The package must have a defining job.

When To Reach For It

A future package should come with a use-case boundary. The reader should know the conditions under which the package becomes the right next reach.

The language here should stay practical:

  • reach for it when the workflow repeats often enough
  • reach for it when the current stack becomes awkward
  • reach for it when stakeholder expectations require a new output form
  • do not reach for it just to replace a stable workflow with novelty

Typical Workflow

Any future package note should describe where it sits in the workflow sequence.

Does it appear during import, cleaning, transformation, visualization, reporting, or project hygiene.

Can it sit cleanly beside the current stack.

Does it simplify the work or create new layers of ceremony.

That workflow positioning matters more than a feature list.

Watch Out For

Every future package should come with one blunt warning.

What does it tempt the reader to do badly.

Where does it add complexity too early.

What kind of misuse is most common.

That warning keeps the appendix honest.

Example You Can Adapt

A good future package entry should read like a decision memo, not like copied documentation. It should include:

  • the problem it solves
  • the point in the workflow where it belongs
  • the reason it is worth adding
  • the condition under which it is not worth adding yet

That makes the appendix expansion-ready without becoming noisy.

Package Comparison Notes

Some packages solve similar problems. This section exists to stop comparison fatigue from becoming another source of delay.

How To Compare Packages Practically

A package comparison should never ask “Which package is objectively better?” The useful question is “Which package fits the workflow this book is trying to protect?”

For example:

  • readr versus rio is not a contest. readr is the cleaner default when the file type is known. rio is useful when file-type variation itself has become friction.
  • readxl versus openxlsx is not a contest either. One is for reading workbooks. The other is for writing them professionally.
  • ggplot2 versus “some specialized chart package” is not a contest unless the current visual need exceeds what ggplot2 plus scales plus forcats can already handle calmly.

Practical Decision Logic

Prefer the package that:

  • keeps the workflow readable
  • reduces repeated effort
  • fits the project’s actual level of complexity
  • can be understood again six weeks later
  • does not require a new stack just to solve one rare problem

Edge Case Or Limitation

Sometimes the default package is not enough. That is fine. But the edge case should be real, not fashionable.

If the file type is exotic, if the reporting layer is growing more sophisticated, or if automation needs to move beyond the current stack, document that clearly. The exception should stay specific.
